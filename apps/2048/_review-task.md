구현된 2048 게임을 **브라우저에서 검증**하고, 코드 품질을 점검하고, 결과를 `review.md`로 작성한다. 문제를 발견하면 **범위 내에서 수정**한다.

- 구현 파일: `/Users/parkcom/study/my-blog/apps/2048/index.html`, `style.css`, `game.js`.

- `spec.md`, 블로그 루트의 `index.html`·`css`·`js`·`posts`·`CLAUDE.md` 등 그 외 파일은 절대 수정 금지.
- 검증 산출물로 `/apps/2048/review.md`를 새로 작성한다(이건 허용).

## 검증 방법
1. 로컬 정적 서버를 띄운다(권장: Claude Preview MCP. `.claude/launch.json`에 `blog` 설정이 이미 있으면 `preview_start`로 재사용 가능. 없으면 프로젝트 루트에서 `python3 -m http.server 8000`).
2. 브라우저로 `http://localhost:8000/apps/2048/`(또는 프리뷰의 해당 경로)에 접속.
3. **콘솔 에러/네트워크 404가 없는지** 먼저 확인.
4. 아래 시나리오를 가능한 한 실제로 동작시켜(키 입력·클릭·필요시 보드 상태 주입) 확인한다. preview_eval로 게임 내부 상태(window에 노출돼 있으면)나 DOM을 읽어 판정해도 좋다.

## 테스트 시나리오 (spec §7)
1. 초기: 타일 2개, 점수 0, best 복원.
2. 기본 병합: `[2,2,_,_]` 한 줄에서 해당 방향 → 4, 점수 +4.
3. **더블 병합 잠금**(가장 중요): `[2,2,2,2]→[4,4]`, `[4,4,2,2]→[8,4]`, `[2,2,2,0]→[4,2]`, `[2,0,2,2]→[4,2]`. 8이 나오면 실패.
4. no-op 게이트: 벽에 붙은 채 같은 방향 키 → 보드·점수 불변, 새 타일 안 생김.
5. 새 타일 분포가 대략 2:4 = 9:1.
6. 승리(2048) 오버레이 + 계속하기 동작.
7. 게임오버: 빈 칸 없음 AND 인접 병합 불가일 때만.
8. best 영속(새로고침 후 유지).
9. 새 게임 버튼: 초기화, best 유지, 오버레이 닫힘.
10. 키보드 방향키 조작 + 방향키로 페이지 스크롤 안 됨.
11. 모바일 스와이프(터치 에뮬레이션 또는 코드 검토로 `{passive:false}`+preventDefault, 임계값, 방향 판정 확인). 짧은 탭 무시.
12. 반응형: 좁은 뷰포트(예: 375px)에서 보드가 화면에 맞고 가로 스크롤·글자 깨짐 없음.
13. 라이트/다크 가독성·대비.
14. 접근성: 점수 `aria-live`, 승패 알림, 보드 포커스, `prefers-reduced-motion`.

## 코드 점검
- 명백한 버그·미사용 코드·전역 오염·접근성 결함이 없는지 game.js를 읽어 확인.
- 특히 4방향 이동 매핑이 각기 올바른지(특정 방향이 다른 방향과 동일 동작하지 않는지) 한 번 더 확인.

## review.md 작성 형식
다음을 포함해 `/apps/2048/review.md`에 작성:
- 검증 환경(URL, 뷰포트).
- 시나리오별 결과 표(통과/실패/비고).
- 발견한 문제와 **수정 내역**(수정했다면 무엇을 어떻게).
- 콘솔/네트워크 상태.
- 남은 위험·후속 권고(있으면).
- 최종 판정: **PASS / FAIL**.
