# Plan 서브에이전트 지침 — 픽셀 아트 에디터

## 역할
너는 **Plan 단계** 서브에이전트다. 구현은 하지 말고, 아래 웹앱의 **상세 구현 계획**만 텍스트로 반환하라(파일 쓰기 권한 없음). 부모가 이를 `spec.md`로 저장한다.

## 만들 웹앱
**픽셀 아트 에디터**
- **16×16 격자**에 도트(픽셀)를 찍어 그림을 그린다. 클릭으로 한 칸 칠하고, 드래그로 연속해서 칠한다.
- **색상 팔레트**: 미리 정의된 색 여러 개 + 현재 선택색 표시. (선택) 커스텀 색 입력(`<input type="color">`)도 검토.
- **PNG 저장**: 완성한 그림을 PNG 파일로 내려받는다. (격자선/UI는 빼고 순수 픽셀만, 선명하게(안티에일리어싱 없이) 적당한 배율로 내보내기.)

## 제약 (프로젝트 규칙)
- 순수 HTML, CSS, JavaScript(ES 모듈)만 사용. 프레임워크·번들러 금지.
- 외부 라이브러리 최소화(가능하면 0). CDN은 필요 시만.
- **모바일 지원 필수**: 터치로도 그릴 수 있어야 함(드래그 페인트, 스크롤/확대 방지 처리).
- 앱은 `/apps/pixel-art/` 폴더 안에서 **자체 완결**. 블로그의 다른 파일에 의존하지 않음. 단, 블로그의 라이트/다크 톤과 어울리면 좋다(독립 구현, 같은 `"theme"` 키 읽기).

## 계획에 반드시 포함할 것
1. **파일 구조** — `/apps/pixel-art/` 안에 둘 파일 목록과 각 역할(예: `index.html`, `style.css`, `editor.js`). spec.md/review.md/지침 파일 제외.
2. **격자/그리기 모델** — 픽셀 데이터 표현(예: 길이 256 색 배열 또는 16×16), DOM 격자(div 16×16) vs `<canvas>` 중 무엇으로 렌더·편집할지와 그 이유. 칠하기/지우기 동작.
3. **입력 처리** — 마우스(다운+이동 드래그 페인트, 격자 밖으로 나갔다 들어와도 자연스럽게) + 터치(스와이프 드로잉, `touch-action:none`/preventDefault로 스크롤·확대 방지). 좌표 → 셀 인덱스 변환 방법.
4. **색상 팔레트 UI** — 기본 색 목록(합리적인 12~16색 제안), 현재 선택색 강조, (선택)커스텀 색·지우개. 선택 상태 관리.
5. **도구** — 최소: 펜, 지우개, 전체 지우기(Clear). (선택) 채우기(fill), 되돌리기(undo) 등은 범위/비범위로 명확히.
6. **PNG 내보내기** — 보이는 격자와 별개로 16×16(또는 확대된) `<canvas>`에 픽셀을 그려 `toDataURL`/`toBlob` → `<a download>`로 저장. 투명 배경 처리 여부, 배율(예: 셀당 16~32px) 권고. `imageSmoothingEnabled=false`.
7. **레이아웃/반응형/테마** — 캔버스 영역 + 팔레트 + 툴바 배치, 모바일에서 격자가 화면에 맞게. 블로그 크림/잉크 톤 라이트·다크 팔레트와 FOUC 방지 인라인 스크립트(같은 `"theme"` 키).
8. **접근성** — 시맨틱 마크업, 버튼은 실제 `<button>`, 색 선택 키보드 접근, 격자 조작 안내, 충분한 대비.
9. **검증 방법** — 로컬 서버로 어떻게 확인할지, 어떤 시나리오(그리기·드래그·지우개·Clear·팔레트 전환·PNG 저장·모바일 터치·반응형·라이트/다크)를 테스트할지.

## 출력 형식
사용자 승인용 `spec.md`에 그대로 담길 수 있도록 제목·불릿으로 구조화한 마크다운으로 반환하라. 간결하지만 구현자(Build 에이전트)가 추가 조사 없이 바로 만들 수 있을 만큼 구체적으로. 범위/비범위도 명시하라.
